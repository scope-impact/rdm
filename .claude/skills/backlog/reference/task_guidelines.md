# Task Creation Guidelines (Reference)

Detailed guidelines for creating, formatting, and breaking down tasks in Backlog.md.

## Title

Clear, brief summary of the task.

## Description

Concise summary of purpose and goal. No implementation details. For subtasks, use user story format: "As a **Role**, I want to **action** so that I can **benefit**."

## Business Value

Why this matters to the project or users. Parsed by rdm from `## Business Value` section.

## Acceptance Criteria

Specific, measurable outcomes using checkboxes (`- [ ]`).

**Good AC principles:**
- Outcome-oriented (result, not method)
- Testable/verifiable
- Clear and concise
- User-focused where applicable

**Examples:**
- Good: `- [ ] User can log in with valid credentials`
- Good: `- [ ] System processes 1000 RPS without errors`
- Bad: `- [ ] Add handleLogin() function in auth.ts`

## Post-Processing Conventions

The CLI (v1.40.0) creates tasks that need manual editing for rdm compatibility:

### 1. Description -> User Story Format

**CLI creates:**
```markdown
## Description

<!-- SECTION:DESCRIPTION:BEGIN -->
Add unit tests for the module
<!-- SECTION:DESCRIPTION:END -->
```

**Convert to** (markers are auto-stripped by rdm, but rewrite the text):
```markdown
## Description

As a **Platform engineer**, I want to **add unit tests** so that I can **catch bugs before deployment**.
```

### 2. Acceptance Criteria -> Separate Lines

**CLI creates** (when using `--ac "First,Second,Third"`):
```markdown
- [ ] #1 First,Second,Third
```

**Use separate `--ac` flags instead:**
```bash
backlog task create "Title" --ac "First criterion" --ac "Second criterion" --ac "Third criterion"
```

### 3. Add rdm-Specific Sections

These sections are parsed by rdm but NOT generated by the CLI:

```markdown
## Business Value

Why this matters to the project or users.

## Subtasks

- FT-042.01: First subtask
- FT-042.02: Second subtask

## Implementation Notes

- **File:** `path/to/implementation`
- **Complexity:** Low/Medium/High
- **Priority:** P0/P1/P2
- **Related stories:** {task_prefix}-XXX
```

### 4. Add `milestone` to Frontmatter

The CLI doesn't generate a `milestone` field. Add it manually:

```yaml
milestone: m-1
```

## Task Breakdown Strategy

1. Identify foundational components first
2. Create tasks in dependency order (foundations before features)
3. Ensure each task delivers value independently
4. Avoid tasks that block each other
5. Use `--dep` flag to declare dependencies

### Requirements

- **Atomic**: Single unit of work, single PR scope
- **Testable**: Clear success criteria
- **Independent**: No dependencies on future tasks
- **Never reference future tasks** (only id < current)

**Correct splitting:**
1. Add system for handling API requests
2. Add user model and DB schema
3. Add API endpoint for user data

**Wrong splitting:**
1. Add API endpoint for user data (should not be first)
2. Define user model and DB schema (should be first)

---

## Quality Checks

Before finalizing task creation:
- [ ] Title is clear and brief
- [ ] Description explains WHY without HOW
- [ ] Each AC is outcome-focused and testable
- [ ] Task is atomic (single PR scope)
- [ ] No dependencies on future tasks
- [ ] Dependencies declared with `--dep` if applicable

---

## Subtasks

**Use FULL parent task ID:**

```bash
# CORRECT
backlog task create -p hh-infra-009 "Subtask title"

# WRONG - creates task-9.01 instead of hh-infra-009.01
backlog task create -p 9 "Subtask title"
```

## Complete Example (Post-Processed)

Shows a subtask after CLI creation + manual post-processing:

```markdown
---
id: HH-INFRA-009.01
title: Add tofu test for K3s module
status: To Do
assignee: []
created_date: '2026-01-18'
labels: [tech-debt, testing, tofu-test, k3s, P0]
dependencies: []
parent_task_id: HH-INFRA-009
priority: high
---

## Description

As a **Platform engineer**, I want to **add unit tests for the K3s module** so that I can **catch cloud-init, IAM, and security group bugs before deployment**.

## Business Value

Prevents infrastructure regressions that could cause production outages.

## Acceptance Criteria

<!-- AC:BEGIN -->
- [ ] #1 Cloud-init template renders valid base64
- [ ] #2 IMDSv2 enforcement validated
- [ ] #3 Root volume encryption enabled
- [ ] #4 All tests pass with `tofu test`
<!-- AC:END -->

## Implementation Plan

1. Research existing test patterns in other modules
2. Create test file `infra/modules/k3s/tests/unit.tftest.hcl`
3. Implement test cases for each AC
4. Run and verify with `tofu test`

## Implementation Notes

- **File:** `infra/modules/k3s/tests/unit.tftest.hcl`
- **Complexity:** High
- **Priority:** P0
- **Related stories:** hh-infra-003.01, hh-infra-003.02
```

### rdm Parser Compatibility Notes

The backlog CLI creates markdown files that rdm's `backlog_parser.py` then consumes:

- **AC markers required:** `<!-- AC:BEGIN -->` / `<!-- AC:END -->` with `- [x] #N text` pattern — the CLI generates these correctly
- **Section headings must be exact:** `## Description`, `## Business Value`, `## Acceptance Criteria`, `## Subtasks` — rdm uses `extract_section()` with exact heading names
- **`<!-- SECTION:DESCRIPTION:BEGIN/END -->` markers** — auto-stripped by rdm parser, no manual removal needed
- **`assignee` and `dependencies` fields** — generated by CLI, ignored by rdm (via `extra="allow"`)
- **`milestone` field** — NOT generated by CLI, add manually to link tasks to milestones
- **`parent_task_id` field** — generated by CLI for subtasks, used by rdm to distinguish subtasks from parent tasks
- **CLI uppercases IDs** — `task_prefix: "ft"` produces `FT-1`, not `ft-1`
