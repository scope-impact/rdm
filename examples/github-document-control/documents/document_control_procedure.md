# Purpose

Defines how controlled documents are created, reviewed, approved, released,
revised, and retained using **git** as the document control system with
**GitHub** as the service provider. Approval is an electronic signature under
21 CFR Part 11; this procedure states how each applicable Part 11 control is
met.

# Scope

All controlled documents in this repository's `documents/` directory. Planning
artifacts (issues, project boards) are coordination records, not controlled
documents, and are never cited as evidence.

# The system

| Document-control function | git / GitHub mechanism |
| --- | --- |
| Master copy | the repository's default branch (`main`) |
| Draft | a feature branch + pull request |
| Review & approval | pull-request review by a code owner (an electronic signature) |
| Release | an annotated, protected git tag + GitHub Release with rendered artifacts |
| Revision history | the git commit history (SHA-chained, time-stamped) |
| Withdrawal | removal via an approved pull request; the history retains the record |

# Controls

## Identity and revision

Every controlled document carries its `id` and `revision` in YAML frontmatter.
The **current approved revision** of any document is its content at the tip of
the default branch; the revision of record for a release is the content at the
release tag. Superseded revisions remain retrievable from history but are never
served from the default branch, so they cannot be mistaken for current.

## Review and approval — electronic signatures

Changes to controlled documents reach the default branch only through a pull
request approved by at least one **independent** code owner (the author cannot
approve their own change). This approval is the electronic signature of record:

- **Manifestation** [[P11:11.50]] — the pull-request review permanently records
  the approver's account (resolvable to their verified name), the UTC
  timestamp, and the meaning of the signing (state `APPROVED`, meaning
  *reviewed and approved for release to the default branch*).
- **Linking** [[P11:11.70]] — the review is bound by GitHub to the exact commit
  SHA reviewed; the merge commit records both. A signature cannot be excised
  from, copied to, or transferred onto other content, because the SHA it signs
  is content-addressed.
- **Uniqueness** [[P11:11.100]] — one GitHub account per individual, single
  sign-on enforced, credential sharing prohibited, accounts never reassigned;
  two-factor authentication is required org-wide.
- **Authority** [[P11:11.10g]] — repository rulesets restrict who may approve
  (code owners) and who may merge; write access is granted per the access
  policy below. Commits must be cryptographically signed (verified signatures
  required by ruleset), so authorship of every change is attributable.

## Access

Repository access is limited to authorized individuals [[P11:11.10d]] through
GitHub organization membership, team-based permissions (least privilege), and
mandatory two-factor authentication. Access grants and removals follow the
personnel on/off-boarding SOP; the GitHub audit log records permission changes.

## Audit trail

The git history is the audit trail [[P11:11.10e]]: every change is a
time-stamped, author-attributed, cryptographically chained commit, generated by
the system on every action. History rewriting and branch deletion on `main`
are blocked by ruleset (non-fast-forward and deletion rules), so later changes
cannot obscure earlier entries; the trail is retained as long as the records.
Pull-request review events (who approved what, when) are retained by GitHub
alongside the commits they sign.

## Permitted sequencing

Rulesets enforce the required order of steps [[P11:11.10f]]: no direct pushes
to `main`; a pull request must exist, required status checks (rendering, gap
analysis) must pass, and a code-owner approval must be current (stale reviews
are dismissed on new pushes) before merging is possible.

## Copies

Accurate and complete copies [[P11:11.10b]] are produced in human-readable and
electronic form: the release workflow renders each controlled document to PDF
via `rdm render` and attaches the artifacts to the GitHub Release; `git
archive <tag>` reproduces the complete electronic document set for any release.

## Retention and retrieval

Records are protected for ready retrieval throughout the retention period
[[P11:11.10c]]: release tags are protected against deletion and movement,
every clone is a full replica of the history, and a scheduled mirror to an
organization-controlled backup provides provider-independent retention. The
retention period for released document sets is defined in the quality manual.

## System validation

The system is validated for its intended use [[P11:11.10a]]: the controls
above are **configuration as code** (`github/` in this example — rulesets,
CODEOWNERS, release workflow), and each is verified by an automated acceptance
test in `tests/acceptance/`, executed on every change. Altered records are
discernible by construction: any content change produces a different SHA, and
unsigned or history-rewriting changes are rejected by ruleset. GitHub itself is
a qualified external service provider (supplier evaluation per the purchasing
SOP); this procedure and its tests constitute the documented validation of the
configuration in use.

## Training

Persons using the system have documented training [[P11:11.10i]] on this
procedure and on git/GitHub basics before receiving write access; training
records are kept per the training SOP.

## Control of this documentation

This procedure and the configuration files it validates are themselves
controlled documents in this repository [[P11:11.10k]] — changes to them
follow the same pull-request approval path they describe.
